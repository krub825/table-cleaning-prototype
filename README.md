# Table Cleaning Detection Prototype

## Описание

Прототип системы детекции состояния столика по видео.
Решение демонстрирует базовый пайплайн обработки: от анализа видеопотока до извлечения событий и расчета метрик.

Цель — определить:

* когда столик **пустой**
* когда столик **занят**
* когда происходит **подход к столику**

## Используемый подход

В рамках прототипа реализована упрощенная логика без обучения моделей.

Используются два сигнала:

1. **Сравнение с эталонным пустым кадром**

   * позволяет определить, насколько текущий вид стола отличается от "пустого"
   * используется для фиксации состояния `occupied`

2. **Детекция движения между соседними кадрами**

   * позволяет фиксировать кратковременную активность (подход, действия руками и т.п.)
   * используется для состояния `approach`

Таким образом:

* `EMPTY` — нет движения и нет значимых изменений относительно пустого стола
* `APPROACH` — есть движение в зоне столика
* `OCCUPIED` — сильное устойчивое отличие от пустого состояния

Для подавления шума используется накопление состояния по нескольким кадрам.

---

## Почему выбран именно этот подход

* не требует обучения моделей
* быстро реализуется и стабильно работает на одном видео
* хорошо интерпретируем (понятно, почему произошло событие)
* соответствует требованиям ТЗ (упрощенный прототип)

Более сложные подходы (например, YOLO) сознательно не использовались, чтобы сохранить простоту и прозрачность решения.

---

## Выбор видео и столика

Использовано видео: Видео 2

Выбран конкретный столик с координатами:

```
x=775, y=4, w=487, h=223
```

Этот кейс выбран не случайно — он содержит несколько сложных ситуаций.

---

## Ограничения и интересные кейсы

В видео присутствуют реальные артефакты, которые влияют на детекцию:

1. **С первых кадров стол уже занят**

   * человек практически не двигается
   * затрудняет определение начального состояния

2. **Столик не полностью в кадре**

   * часть области выходит за пределы видео
   * влияет на стабильность детекции

3. **Посетитель оставляет поднос до того, как сесть**

   * система может трактовать это как "занято"
   * но:

     * не всегда корректно учитываются оставленные вещи без движения
     * это ограничение текущего подхода

4. **Окно рядом со столом**

   * изменение освещения (солнце)
   * приводит к ложным изменениям пикселей

   Ограничения:

   * не учитываются условия день/ночь
   * не компенсируется изменение освещения

## Аналитика

Рассчитывается время между освобождением столика и следующим подходом:

```
empty_time  = 109.667
approach_time = 184.067
delay = 74.4 сек
```

Среднее значение:

```
74.4 секунды
```

## Вывод

Решение демонстрирует полный пайплайн:

* выделение ROI
* детекция событий
* логирование
* расчет метрик
* визуализация

При этом остаются ограничения, связанные с:

* изменением освещения
* частичной видимостью стола
* отсутствием семантической информации (например, распознавания объектов)
